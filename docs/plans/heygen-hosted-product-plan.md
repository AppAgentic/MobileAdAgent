# Mobile Ad Agent Product Plan

Last updated: 2026-06-23

> Scope correction (2026-06-23): HeyGen is used **only** as a hosted HyperFrames
> render/edit backend for the deterministic finishing stage. It is **not** the UGC
> generator. All UGC generation is **our own pipeline** — the cloud productization
> of the PepMod-style flow we validated locally (script/brief → creator-style
> generation → proof cutaways/b-roll → multimodal QA → HyperFrames finishing).

## Decision

Mobile Ad Agent is **our own proof-driven UGC creative factory**. It is the cloud
productization of the local pipeline that produced the PepMod UGC: we own the brief,
script, creator generation, proof cutaways, QA, manifests, cost ledger, and artifacts.

The only thing we rent from HeyGen in V1 is the **hosted HyperFrames renderer** for the
finishing/editing stage (hook + CTA captions, proof overlays, thumbnails, contact
sheets). We rent it purely so we do not have to stand up our own HyperFrames Cloud Run
worker on day one. Our benchmark already showed a HyperFrames worker can reproduce that
finishing stage, so HeyGen-hosted HyperFrames is a temporary convenience, not a
dependency anything else relies on.

### What HeyGen is and is NOT here

- **IS:** the hosted HyperFrames Cloud renderer for the deterministic finishing stage
  only — captions, proof/overlay compositing, CTA, first-frame/thumbnail, contact
  sheet. One `RenderBackend` among several.
- **IS NOT:** the UGC/talking-head generator. We do **not** use HeyGen Direct Video,
  HeyGen Video Agent, or HeyGen Cinematic Avatar for the product's creative generation.
  Creator generation is our own pipeline (Nano Banana identity transform → Kling/Seedance
  image-to-video with native audio → ffmpeg/Gemini QA), exactly like the PepMod flow.

So HeyGen sits behind exactly one swappable interface (`RenderBackend = heygen-hyperframes-cloud`)
and is replaced in Phase 2 by `hyperframes-cloud-run`. Generation never touches HeyGen.

## Why this shape

We just proved two things locally on branch `implement-local-renderer-benchmark`:

1. Our own pipeline already makes the actual creative (the PepMod UGC: script → creator
   video → proof cutaways → captions). That is the product's core competence and we keep it.
2. The finishing/render layer is portable: a HyperFrames composition can closely reproduce
   the Remotion AdRenderer caption-finishing (SSIM 0.977 on the PepMod copy, lint clean).
   So "hosted HyperFrames now, our own HyperFrames worker later" is a low-risk swap.

Therefore: own the hard, defensible part (the UGC quality pipeline) from day one; rent
only the easy-to-replace finishing renderer (HyperFrames) until we move it onto Cloud Run.

## Product Shape

Mobile Ad Agent is an app-proof ad creative factory.

The customer brings:

- App Store / Play Store URL
- app icon
- app screenshots or screen recordings
- product facts and avoid-list
- target audience and channel
- optional preferred creator/reference style
- optional existing winning ad examples

The product returns:

- proof-driven UGC videos (generated by our pipeline)
- finished image ads / static variants
- thumbnails / first frames
- captions and CTAs
- QA reports
- contact sheets
- downloadable creative packs
- API/MCP-readable job state

The product does not directly mutate Meta, TikTok, Google Ads, Apple Search Ads, or live
budgets in V1.

## The Generation Pipeline (our own — the PepMod flow, productized)

This is the part we own and never delegate to HeyGen. It mirrors the local
`ugc-script-creator` → `ugc-video-producer` chain, but behind durable state, cost caps,
and worker queues.

1. **App Intake**
   - User creates an app profile or bootstraps from store URL.
   - Store icon, screenshots, product facts, audience, compliance notes, supported claims,
     and avoid-list.
   - Classify screenshots as raw proof, stylized store screenshot, rawifiable, or unusable.

2. **Proof Library**
   - Upload raw app screenshots, screen recordings, app icon, and store screenshots.
   - Extract proof objects: score cards, dashboards, protocol screens, converters,
     rankings, logs, charts, before/after alternatives, etc.
   - Every proof object stores source asset, crop bounds, OCR text, trust level, supported
     claims, and visual category. No fabricated app UI ever enters this library.

3. **Creative Brief** (Creative Director Agent)
   - Picks format family, proof object, hook angle, target duration, caption plan, and
     acceptance criteria. Provider-neutral and auditable.

4. **Script + Scene Plan** (Script Agent)
   - Exact spoken script, hook caption, CTA caption, proof-cutaway timings, claim
     references. QA verifies only supported product facts are used (no invented features).

5. **Creator Generation — OUR pipeline** (Generation Worker)
   - Identity transform of a reference/creator image with Nano Banana (mandatory identity
     change, never ship a reference subject's likeness).
   - Image-to-video per segment with **Kling v3 Pro** (native audio) or one-shot
     **Seedance**, with exact-script enforcement and static-camera negative prompts.
   - ffmpeg silencedetect dead-zone trimming for tight cuts.
   - This stage is provider-neutral behind `GenerationBackend` (Kling, Seedance, fal,
     future owned models) — **HeyGen is not an option here**.

6. **Proof Cutaways / B-roll**
   - Splice cropped real app proof clips/screenshots into the creator video at the
     planned timings (the "demo proof" beats), produced from the proof library — never
     fabricated UI.

7. **Finishing — HeyGen hosted HyperFrames (V1), then our Cloud Run HyperFrames**
   - Compose hook caption, CTA, proof/overlay finishing, first-frame thumbnail, app icon,
     optional ducked music via a `RenderBackend`.
   - V1 backend: `heygen-hyperframes-cloud`. Phase 2 backend: `hyperframes-cloud-run`.
     Remotion AdRenderer (`remotion-cloud-run` / `remotion-lambda`) kept as a fidelity
     fallback for templates HyperFrames cannot yet match.
   - Store the HyperFrames composition zip, input variables, output MP4, contact sheet,
     and report in R2.

8. **QA** (QA Agent)
   - ffprobe dimensions, fps, duration, codec, file size.
   - OCR hook/CTA/proof text; verify proof legibility and no fake UI.
   - Verify claims against product facts; verify creator/audio/script alignment.
   - Gemini/multimodal QA for artifacts, dead zones, bad hands/phones, uncanny
     presentation, caption scale/units (SSIM alone missed a caption-scale bug — always
     inspect the contact sheet).
   - Hold or retry with a structured reason instead of silently shipping weak assets.

9. **Pack + Export**
   - Package MP4s, first frames, thumbnails, QA reports, manifests, copy variants, platform
     notes. Downstream handoff is a separate explicit export, never an ad-network mutation.

## Backend Boundaries

### Control Plane

- Next.js on Firebase App Hosting
- Firebase Auth for users, teams, roles
- Firestore for canonical state
- Cloudflare R2 for the binary artifact lake
- Cloud Run workers for generation, proof-cutaway editing, ffmpeg, finishing, QA, and
  provider calls

### Provider Secrets

No provider key (HeyGen, fal, Kling, Gemini, etc.) in browser code, Firestore documents,
prompt text, Slack summaries, or committed files. Server-side Secret Manager / vault paths
only, e.g.:

```text
providers/heygen/api-key        # hosted HyperFrames finishing only
providers/fal/api-key           # generation
providers/google-ai/api-key     # Nano Banana / Gemini QA
```

Workers read keys at runtime and call providers from the backend only.

### Provider State

Provider URLs are transient outputs, not canonical storage. On completion:

1. receive webhook or poll status
2. download the media server-side
3. write to R2
4. store the R2 object key and metadata in Firestore
5. mark the provider URL as external provenance only

### Why This Matters

`creativeJobs` and the UI never know which model made the talking head or which renderer
finished it. The job knows: exact script, creator constraints, proof objects used, output
asset IDs, cost, QA result. Provider-specific request JSON lives in a provider manifest
stored as an artifact. Because finishing is behind `RenderBackend`, swapping
`heygen-hyperframes-cloud` for `hyperframes-cloud-run` changes nothing upstream.

## HeyGen HyperFrames Integration (the only HeyGen surface)

- Use HeyGen's hosted **HyperFrames Cloud** endpoint to render a self-contained
  HTML/CSS/JS composition bundle to MP4/WebM/MOV.
- Inputs: the finished creator video + proof cutaways (from our pipeline) plus the
  HyperFrames composition zip and variables (captions, CTA, overlays, thumbnail hold).
- Server-side only via `X-Api-Key`; base `https://api.heygen.com`. Download result into R2.
- Official pricing checked 2026-06-23: HyperFrames 1080p/30fps is billed at
  `0.1 credits/min` of output. Convert credits to dollars from the active HeyGen account
  plan before showing customer margin; at a `$1/credit` assumption, a 15s render is about
  `$0.025`.
- Migration target: `hyperframes-cloud-run` worker running HyperFrames locally with the
  identical composition zip/variables contract (benchmark already validated parity).

## Cost Model

Two cost surfaces, owned separately:

| Stage | Backend (V1) | Approx unit cost |
| --- | --- | --- |
| Creator generation (our pipeline) | Kling v3 Pro i2v | ~`$9`/UGC video (multi-segment, native audio) |
| Creator generation (alt) | Seedance i2v | ~`$0.30/sec` (15s ≈ `$4.50`) |
| First frames / static ads | Nano Banana 2 | per-image |
| Finishing | HeyGen HyperFrames Cloud | `0.1 credits/min` at 1080p/30fps (15s = `0.025` credits) |

Generation dominates cost, so cost ceilings live on `generationTasks`. Finishing is cheap
either hosted or on Cloud Run, which is exactly why HeyGen-hosted finishing is acceptable
short-term and trivial to bring in-house.

Enforce per-job estimate before submit, workspace monthly cap, hard retry ceiling, at most
one automatic expensive regeneration per held creative, and an operator-visible cost ledger.

## Roll-Our-Own Migration Plan

We already own generation. Migration is mostly about owning the finishing renderer and
broadening generation provider choice. Hosted V1 must already write the same specs and
artifacts the self-hosted finishing worker will consume.

### Phase 1: Product Fit

Own pipeline for generation (Nano Banana + Kling/Seedance + proof cutaways + QA) with
HeyGen-hosted HyperFrames for finishing; R2 + Firestore as source of truth.

Exit criteria:

- users can create an app, upload proof, generate a pack, review QA, download assets
- 80%+ of generated assets are passable or repairable
- true cost per approved creative is understood
- at least one repeatable creative family works across multiple apps

### Phase 2: Own The Finishing Renderer (retire hosted HeyGen)

Move HyperFrames finishing from HeyGen-hosted Cloud to our own `hyperframes-cloud-run`
worker, keeping identical composition zips/variables. Prioritize caption overlays, proof
overlays, thumbnails/first frames, contact sheets, ffprobe, OCR, audio normalization.
This is the lowest-risk migration — the benchmark already showed HyperFrames reproduces
PepMod-style caption finishing. After this, HeyGen is no longer in the product path.

### Phase 3: Broaden Generation Providers

Provider-neutral `GenerationBackend` adapters for fal.ai / Seedance / Kling variants and
Gemini/OpenAI image generation for first frames and static ads. UI/public API unchanged.

### Phase 4: Own UGC Generation Quality Loop

Productize the `ugc_video_producer` quality loop behind durable state and cost caps:
reference creator workflow, exact-script enforcement, dead-zone trimming, proof cutaway
timing, multimodal QA, automatic repair loops.

### Phase 5: Optional Model Ownership

Only if volume and strategic moat justify it. Until then, owning prompts, proof assets, QA,
and finishing logic matters more than owning raw model inference.

## Key Product Rules

- No fake app UI.
- No unsupported claims.
- No live ad-network mutations.
- No provider keys in client code.
- No canonical storage of signed provider URLs.
- HeyGen is only ever a `RenderBackend` (finishing), never a `GenerationBackend`.
- Every expensive task has an estimate, actual cost, retry count, and ceiling.
- Every output has a manifest, contact sheet, first frame, ffprobe data, QA verdict, and
  source proof IDs.
- Every generated UGC asset must remain editable/recreatable from stored provider-neutral
  specs.

## Recommendation

Ship the first usable product with:

- **Our own UGC generation pipeline** (Nano Banana identity → Kling/Seedance i2v native
  audio → proof cutaways → multimodal QA) — the productized PepMod flow.
- **HeyGen HyperFrames Cloud** for hosted finishing only, behind `RenderBackend`.
- **R2 + Firestore** as the only source of truth.
- **Provider-neutral generation + render task schemas** from day one.
- **Self-hosted HyperFrames on Cloud Run** as the first migration target, after which
  HeyGen leaves the product path entirely.

This keeps the defensible part (UGC quality) in-house from the start and rents only the
cheap, already-benchmarked finishing renderer until we move it onto Cloud Run.
